#1 - Agent Core Engine Overhaul
What & Why
Overhaul the core agent engine to fix critical reliability issues causing repeated errors, lost file paths, and failed executions — the exact failures shown in the screenshot. Restructure the agent loop, E2B sandbox integration, and shell execution to match ai-manus architecture: isolated per-session workspace, persistent shell sessions, and fail-safe error recovery.

Done looks like
Agent never runs python script.py without first verifying the file exists in the same directory
Files written via file_write are immediately and reliably available in E2B shell (no "No such file or directory")
Agent detects and breaks out of repeated error loops within 2 attempts, switching approach automatically
Shell sessions are persistent across multiple tool calls (same session ID = same shell state)
E2B sandbox workspace /home/user/dzeck-ai/ persists reliably across all tool calls in a session
Agent recovers from E2B sandbox timeout/restart without losing written files (file cache replay works)
Context window never overflows — memory is compacted before hitting token limits
Each user session gets its own isolated workspace subdirectory
Out of scope
Switching LLM provider (stays on Cloudflare Workers AI)
Frontend UI changes (separate task)
Prompt text rewrites (separate task)
Tasks
Fix E2B file sync — After every file_write call, immediately sync the file into the active E2B sandbox via the sandbox filesystem API (not just in-memory cache). Add a post-write verification step that confirms the file exists in E2B before returning success. Fix the file cache replay so it runs reliably when a sandbox is recycled.

Add pre-execution file checks — In shell_exec, detect when the command is python script.py or python3 /path/to/file.py (or similar patterns). Before running, check if the target file exists in E2B; if not, return a clear error message instructing the agent to write the file first instead of silently failing with "No such file or directory".

Strengthen repeated error detection — The current _check_repeated_error function exists but isn't stopping the loops reliably. Tighten it: if the EXACT same command (or same error signature) appears twice, immediately block execution and return a structured error response with the message "BLOCKED: identical command/error seen before — change approach entirely". Track this at the session level, not just within a tool call.

Persistent shell sessions across tool calls — Refactor ShellTool so that when shell_exec is called with the same id, it reuses an existing persistent subprocess (currently using threading.Lock per session). Fix state tracking so shell_view correctly reads buffered output from background processes and shell_wait properly blocks until the process emits output matching a pattern or times out.

Per-session workspace isolation — Namespace the E2B workspace by session ID: /home/user/dzeck-ai/<session_id>/ instead of shared /home/user/dzeck-ai/. Update all workspace path references in the tools and prompts accordingly. This prevents cross-session file contamination.

Context memory compaction — Add automatic context compaction in execute_step_async: when the accumulated exec_messages list exceeds 12 messages, summarize the middle messages into a single compressed context message, keeping only the system prompt, first user message, last 4 messages, and the compressed summary. This prevents token limit errors on long tasks.

Better agent error recovery in the main loop — In run_async, when a step fails (ExecutionStatus.FAILED), don't immediately mark the whole plan as failed. Instead, retry the step once with an explicit error context injected into the execution prompt ("Previous attempt failed with: [error]. Take a different approach."). Only fail the plan after 2 consecutive failed attempts on the same step.
