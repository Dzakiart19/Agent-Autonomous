Plan #2 - Fix: Agent Berhenti & Lanjut Setelah message_ask_user
#2 - Fix: Agent Berhenti & Lanjut Setelah message_ask_user
What & Why
Dua bug kritis ditemukan saat testing agent mode:

Bug 1 — Semua plan step ter-ceklis sekaligus: Saat message_ask_user dipanggil di step 1, fungsi _run_tool_streaming memang yield __step_done__, tapi while-loop utama di run_async tidak punya pengecekan apakah eksekusi harus berhenti (waiting_for_user). Akibatnya loop terus ke step 2, 3, 4, dst., semua step langsung dieksekusi dan dikirim ke frontend — muncul checklist semua sekaligus.

Bug 2 — Setelah user balas, agen jadi chat biasa (tidak lanjut plan): Dua penyebab:

Di run_async, balasan user yang pendek (e.g., "mp3, 320kbps") lolos ke _is_simple_query() karena ≤6 kata → dibalas sebagai teks biasa, bukan lanjutkan plan.
Di ChatPage.tsx, saat user kirim pesan baru, state planMsgIdRef, currentPlanRef, stepHistory langsung di-reset, sehingga konteks plan hilang dari UI.
Done looks like
Saat agent memanggil message_ask_user, eksekusi step berhenti setelah step tersebut — step berikutnya TIDAK dieksekusi atau di-ceklis.
Plan card di UI menunjukkan: step 1 sudah selesai (ceklis), step 2–N masih pending/abu-abu.
Saat user membalas, agent melanjutkan dari step berikutnya dengan konteks plan yang sama, menggunakan tools (bukan teks biasa).
Plan card di UI tetap terlihat dan update step berikutnya satu per satu saat dieksekusi.
Out of scope
Perubahan LLM/model backend
Fitur "rollback" atau step ulang
Perubahan tools selain message_ask_user
Tasks
Hentikan while-loop setelah message_ask_user di run_async — Setelah execute_step_async selesai untuk satu step, cek apakah step tersebut menghasilkan waiting_for_user. Jika ya: simpan snapshot plan ke session (Redis/MongoDB) dengan flag waiting_for_user: True + simpan pending_steps yang belum dieksekusi, lalu break keluar dari while-loop tanpa meneruskan ke step berikutnya.

Resume plan saat user membalas — Di run_async, setelah load session, cek apakah ada flag waiting_for_user: True di session. Jika ada: bypass _is_simple_query sepenuhnya, load plan dari session, inject balasan user sebagai hasil dari message_ask_user step (tambahkan ke exec_messages sebagai user reply), dan lanjutkan eksekusi dari step berikutnya (bukan dari awal).

Pertahankan konteks plan di frontend saat user membalas — Di ChatPage.tsx, saat isWaitingForUser = true dan user mengirim pesan, jangan reset planMsgIdRef, currentPlanRef, dan stepHistory. Kirim request dengan session_id yang sama (sudah dilakukan) tapi tambahkan flag is_continuation: true agar backend tahu ini balasan ke message_ask_user, bukan request baru.

Sinkronisasi flag is_continuation di backend — Di routes.ts, parse is_continuation dari request body dan teruskan ke Python agent subprocess sebagai argument/env var. Di run_async / agent_flow.py, terima flag ini dan gunakan untuk skip _is_simple_query check bila True.
